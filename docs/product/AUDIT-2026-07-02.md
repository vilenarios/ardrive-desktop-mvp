## 1. Money paths (uploads, Turbo, payments)

1. **AR vs Turbo payment selection is cosmetic; every upload executes via Turbo.** Every `arDriveFactory()` call passes `turboSettings` (wallet-manager-secure.ts:181-187, 318-325, 439-446, 1279-1286); ardrive-core fixes `useTurbo: !!turboSettings` at factory time (ardrive_factory.js:34-38). The per-upload flags at sync-manager.ts:2279-2285 (`paymentMethod`/`useTurbo`/`skipBalanceCheck`) are not part of `uploadAllEntities`' signature and are silently ignored. AR balance checks (main.ts:2336-2342, hardcoded 1 winston/byte) validate the wrong currency; DB `uploadMethod` is fiction.
2. **Cancel doesn't abort; retry double-charges.** `uploads:cancel` (main.ts:2399-2424) marks DB failed; `UploadQueueManager.cancelUpload` only removes `'pending'` items (UploadQueueManager.ts:140-149); no AbortController exists. `uploads:retry` (main.ts:2426-2458) resets any row without status check → two concurrent paid uploads of the same file. A "cancelled" in-flight upload that completes flips the record back to `completed` (sync-manager.ts:2671-2698).
3. **All USD prices fabricated**: `MOCK_AR_PRICE_USD = 6.50` (src/renderer/utils/ar-price-utils.ts:13), consumed throughout UploadApprovalQueueModern (202-203, 443, 466, 490, 650-651).
4. **AR cost estimate is hardcoded** 1 winston/byte (CostCalculator.ts:14-16), excludes ~15% community tip; on Turbo error a fake `estimatedCost * 1.1` is displayed as a Turbo quote (CostCalculator.ts:56-66). Turbo pricing itself is live and real (turbo-manager.ts:93-104).
5. **Batch approve**: "Approve & Upload" calls `onApproveAll()` **then** loops per-file approve (Modern.tsx:906-928) — redundant, and files skipped by approve-all's balance check get approved by the loop anyway (single-approve has no AR balance check, main.ts:2179-2229). Approve-all ignores per-file method choice (main.ts:2318 uses `recommendedMethod || 'ar'`). The "Payment Method" radio (`selectedPaymentMethod`, Modern.tsx:70, 772-803) is read by nothing.
6. **Conflict resolution is a no-op**: `handleResolveConflict` only console.logs (Dashboard.tsx:686-688); upstream `conflictType` is hardcoded `'none'` with TODO (sync-manager.ts:1931-1933).
7. **Metadata ops hide/unhide/delete throw** "not yet implemented" (sync-manager.ts:3249-3253) yet the UI offers them (Modern.tsx:36-43); move/rename always use `*Public*` APIs (sync-manager.ts:3228-3243) — fail on private drives.
8. **Payment window**: URL check is only `startsWith('https://')` (main.ts:2798) — no host allowlist; success detection only `will-navigate` + `includes('app.ardrive.io')||includes('success')` (main.ts:2818-2830) — server redirects missed, false positives possible; no `closed` handler → manual close produces no event. No `sandbox: true`, no `setWindowOpenHandler`. Focus-based refresh (main.ts:244-266) matches URLs that never occur — dead.
9. **Queue concurrency**: `setInterval(async, 1000)` with no reentrancy guard (UploadQueueManager.ts:64-66) starts a new upload every second unboundedly; two files in one new folder can double-create the paid folder entity (check-then-create race, sync-manager.ts:2380-2445).
10. **Approval/modification race**: balance validated against size at detection (main.ts:2189) but upload wraps disk state at upload time (sync-manager.ts:2247) — file grown after approval charges more than approved.
11. **Auto Top-Up is deceptive**: `handleAutoTopUpSave` is a TODO + console.log that then shows "saved" (TurboCreditsManager.tsx:213-218).
12. **Free-tier (<100KB) logic WORKS** end-to-end (CostCalculator.ts:4,85-87; main.ts:2183-2186; server-side subsidy ~105KiB covers the `<` vs `<=` off-by-one).
13. Upload execution via `arDrive.uploadAllEntities` streams (no whole-file buffering); TurboManager's Buffer-based `uploadData`/`uploadFile` (turbo-manager.ts:138-211) is dead code. Progress is binary 0→100 (sync-manager.ts:2119-2129); the queue fabricates animated progress for free files (Modern.tsx:237-250).
14. AR→Credits conversion works (main.ts:2604-2625); immediate "Success!" contradicts the UI's own "5-15 minutes" warning; `validateTurboAmount` (1e-12..1,000,000) shared between USD and AR amounts — no meaningful bounds.
15. Offline: AR balance errors swallowed → displayed as **0 AR** with no error indicator (wallet-manager-secure.ts:516-519).

## 2. Sync engine

1. **Local file edits never re-upload.** Path-based dedup bails before the update path: sync-manager.ts:1846-1855 ("File already uploaded, skipping") — only brand-new paths queue. `versionManager.detectFileChange` correctly detects 'update' but is dead-ended.
2. **Failed downloads marked synced.** `downloadFile`'s catch swallows without rethrow (DownloadManager.ts:564-571) → caller sets `syncStatus='synced', localFileExists=true` (1116-1120). Retry/permanent-error logic (1137-1170) unreachable. Hash verification never runs — compares against never-populated `dataContentHash` (DownloadManager.ts:700-705); mismatch would only warn.
3. **No conflict handling anywhere**: downloads filter on existence only (DownloadManager.ts:370-374); final rename overwrites unconditionally (StreamingDownloader.ts:283).
4. **Deletes dead-end**: `FileOperationDetector.onFileDelete` caches into a map nothing reads (`getRecentOperation` zero callers, FileOperationDetector.ts:486-502); ArFS delete throws "not yet implemented" (sync-manager.ts:3250-3253); folder delete is DB-only (1734).
5. **Multi-drive is a facade**: exactly one `new SyncManager` (main.ts:97) with scalar driveId/folder (sync-manager.ts:55-57). Boot and `sync:start` sync only `find(m => m.isActive) || [0]` (main.ts:157, 1817-1821). `multi-sync:*` preload methods (preload.ts:302-309) have **no handlers**. Schema is multi-drive ready (mappingId FKs everywhere); engine isn't.
6. **Stop→start self-corrupts**: `stopSync` destroys `downloadManager` and `progressTracker` (sync-manager.ts:346-349) which are constructed once (81, 93) and never rebuilt; `SyncProgressTracker.destroy()` kills the flush interval while `pendingEmits` keeps queueing (SyncProgressTracker.ts:39-64) — throttled progress never reaches UI again after any stop/switch.
7. **No crash recovery**: no startup status reset anywhere; rows stuck `uploading`/`downloading` stay stuck (re-queue picks only `'pending'`, DownloadManager.ts:428-429); queues are memory-only. No CHECK constraints on uploads/downloads/pending status columns (database-manager.ts:87,143,244).
8. **Folder-vs-drive divergence**: `sync:start` validates against `primaryMapping.localFolderPath` (main.ts:1844) but watches `config.syncFolder` (main.ts:1797); `drive:switchTo` (main.ts:1682-1740) never updates `config.syncFolder` → after a switch, sync can watch one folder while targeting a different drive.
9. Move/rename: folder rename/move to ArFS WORKS (public) (sync-manager.ts:1407-1421, 1562, 1611); file move/rename works via approval (3389-3410 → 3228/3241); move to not-yet-uploaded parent silently dead-ends (TODO, 3377-3381).
10. Robustness: `ErrorHandler.ts` is dead code (zero importers); no offline detection/reconnect; metadata sync failures swallowed "continuing anyway" (889-898); watcher error handler logs only (327-329); `handleFileChange` lacks the state guards `handleFileAdd` has (1296-1328); several un-awaited async watcher callbacks.
11. Size limit: comment says 100MB (sync-manager.ts:1809) but `MAX_FILE_SIZE = 500MB` (CostCalculator.ts:5); oversized files **silently skipped** (1811-1813).
12. Perf: hashing reads whole files via `fs.readFile` up to 3× per event + once post-upload (1744→3203, 1816, 2751); `getProcessedFiles()` loads the entire table per file event (1826, 1936, 1122, 1138).
13. `sync:manual` is download-only despite the name (main.ts:2073-2078); no periodic remote polling while monitoring.
14. FileStateManager 30s "recently downloaded" eviction (FileStateManager.ts:20-23) — downloads finalizing >30s can be re-detected as local adds (feedback loop).
15. Dead code in sync-manager.ts (no call sites): `scanDownloadedFiles` (523), `downloadFilesViaDirectArweaveQuery` (594), `downloadDriveFile` (901), `scanExistingFiles` (1203), duplicate `syncDriveMetadata` (2793), `recursivelyListDriveContents` (2968), `fetchDriveContentsViaGraphQL` stub (3008), `downloadMissingFiles` (3016), `createAllFolders`/`downloadMissingFilesWithProgress` (3063, 3105), `verifySyncState` (3154). Orphans: FileHashVerifier.ts, sync/constants.ts.
16. StreamingDownloader genuinely streams with atomic rename + orphan temp cleanup at startup (StreamingDownloader.ts:119-283; sync-manager.ts:3459-3481). Single hardcoded gateway `arweave.net`; no resume/Range support.

## 3. Private drives

1. **Downloads write ciphertext.** Private *listings* decrypt (listPrivateFolder with key, DownloadManager.ts:743-761) but file content always fetched raw from `https://arweave.net/{dataTxId}` (DownloadManager.ts:586-591, 1023-1024) with no decryption path anywhere in src/main (no downloadPrivateFile/fileDecrypt callers).
2. **Wrong passwords unlock "successfully."** `deriveDriveKey` is pure HKDF — never fails; no trial decryption in `unlockDrive`/`drive:unlock`/`unlockPrivateDrive` (drive-key-manager.ts:81-110; main.ts:809-878; wallet-manager-secure.ts:1229-1249). Garbage key cached; downstream decrypts silently fail (`safelyDecryptToJson` swallows).
3. **Private drive creation: user pays, UI reports failure.** `drive:create-private` returns `{success, data}` (main.ts:801-802) but CreateDriveModal checks `drive.id` on the wrapper (CreateDriveModal.tsx:101-107) → always "Failed to create drive" after the on-chain spend; no mapping created. Public `drive:create` returns raw drive (main.ts:786) — inconsistent envelope is the root cause.
4. **Key-persistence WIP (uncommitted) is inert**: `Buffer.from(key as any)` on an EntityKey **throws** (drive-key-manager.ts:141; verified empirically) → save always fails → unlock with "remember" checked returns false on a *correct* password (drive is actually unlocked in cache — state divergence); `drive-keys.enc` never written so `loadPersistedKeys` (170-208) finds nothing; load path caches raw Buffer where EntityKey expected (193-194) → `driveKey.keyData` undefined → all decrypts fail. Correct approach: `key.keyData.toString('base64')` / `new EntityKey(Buffer.from(...))` + record driveSignatureType. DB prefs table is write-only (`getDriveKeyPersistence`/`getAllPersistedDriveIds` zero callers, database-manager.ts:1628-1676). Plan steps 5 (session-restore in App.tsx) and 6 (settings UI; `drive.isPersisted`/`setPersistence` never called from renderer) unimplemented.
5. **App.tsx drops the persist flag**: modal passes `(password, persistKey)` (PrivateDriveUnlockModal.tsx:8,31) but `handlePrivateDriveUnlock(password)` ignores arg 2 (App.tsx:478-483). DriveSelector forwards it (DriveSelector.tsx:62-66) — into the broken save.
6. **DriveSelector treats unlock response object as boolean** (DriveSelector.tsx:66-72) — `{success:false}` truthy → failure selects drive and closes modal.
7. Private **upload works** (driveKey attached, sync-manager.ts:2264-2272, 2188-2196, 2538-2545; locked → clean failure). Locked drive during sync: listing throws, catch swallows → treated as empty drive, no user-visible error (DownloadManager.ts:749, 842-845). Boot auto-sync has no lock check (main.ts:148-176; manual `sync:start` guards at 1838-1840).
8. Persistence-at-rest design is sound (AES-256-GCM + scrypt via writeEncryptedFile, session password fetched per-call from keychain). `clearAllKeysAndStorage` dead branch: `clearAllKeys()` nulls profileId before the `if (this.currentProfileId)` file delete (drive-key-manager.ts:290-311).
9. Cleanup paths correct: logout/profile-switch clear key cache; profile deletion removes `drive-keys.enc`; `drive:switchTo` locks other private drives (main.ts:1728-1736). `drive:lock` keeps the persistence preference — persisted drives will silently re-unlock post-fix (design decision to revisit).
10. `drive:unlock` runs drive passwords through `validatePassword` (min 8 chars, input-validator.ts:36) — drives created in other clients with shorter passwords can never be unlocked here. No password-change flow exists app-wide. Unlock is unguarded against profile switch mid-modal (keys derived with whatever walletJson is current).
