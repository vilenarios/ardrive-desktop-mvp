Evidence for each item lives in [AUDIT-2026-07-02.md](./AUDIT-2026-07-02.md) (§section.finding). Line numbers there are as-of-audit and may drift.

**Phase/Track** — Phases 1–4 = beta critical path (see [ROADMAP.md](./ROADMAP.md)); Tracks A–E = post-beta.

Working an item: set `in-progress`, implement, verify per the acceptance criteria, set `done` **in the same PR** with a one-line note + commit ref. Never delete items; mark `wont-fix` with a reason.

**Findings filed (real misclassifications caught by testing real behavior; all flagged for the coordinator, none fixed — each is inherent to the detector design and a risky change, not a small safe fix):**
- **F1 (file, medium — money/history): add-before-unlink surfaces a MOVE as a COPY.** If the OS/chokidar emits the new file's `add` before the old file's `unlink`, there is no pending delete yet; the original is still in `processed_files` with the same hash, so `detectCopy` fires and the operation reads as `copy`. sync-manager treats copy as new (handleFileWithVersioning ~:2045 "Handle as new file for now") → the moved content is RE-UPLOADED, and the later `unlink` confirms a delete/hide of the old path. Only bites when the add precedes the unlink (not the common atomic-rename ordering, but not guaranteed across dirs/filesystems). Fix is a restructure (defer copy-vs-move until the window settles), hence flagged not fixed.
- **F2 (folder, medium — history/identity): a new folder cross-matches the FIRST pending delete in the same parent.** `onFolderDelete` builds its snapshot from last-known state with an EMPTY children list and EMPTY contentHash, so `detectOperation` can only compare parent+name. Same-parent+different-name always yields `rename`, and `onFolderAdd` returns on the first non-`new` match — so if two folders are deleted in one parent within the 2s window, a new folder is attributed to whichever delete was recorded first (Map insertion order), i.e. potentially the WRONG ArFS folderId gets renamed. Needs content captured at delete time to disambiguate.
- **F3 (folder, low-medium): a renamed+moved NON-empty folder → `new` (folderId not carried forward).** Because the delete snapshot has empty children/contentHash, the `rename_and_move` branch (needs contentMatch OR childrenSimilarity>80) only fires when BOTH folders are empty; a non-empty renamed+moved folder fails both and is classified `new`, re-creating the folder and orphaning its ArFS identity.
- **Observation (not a live bug): `detectByMetadata` (name+size fallback → `move` without a hash match) is effectively inert in production** — the post-delete snapshot's size is 0 (the file is already gone when `onFileDelete` stats it), so it can't match the new file's real size. It only triggers if the delete stat happens to win a race. Pinned both ways (size captured → `move`; realistic size 0 → `new`).

### FEAT-4 · P2 · Track B · `idea` (post-beta)
**Proof / "anchor" mode — metadata-only, no data upload (Phil, 2026-07-05; D-028).** Write only the ArFS file *metadata* transaction carrying the file's content hash (SHA-256) + name/size/time, and SKIP the data transaction. The file's bytes stay local; the on-chain metadata is a timestamped, immutable *proof the file existed* — not a stored copy. Feasibility is high: ArFS already separates metadata from data (the metadata tx references a data tx id), so this is a data-tx-less variant, not a protocol fork. Metadata is tiny (<105 KiB) → **free via Turbo**. Value: (1) free/near-free proof-of-existence + integrity + provenance (legal/IP/notarization); (2) privacy — only the hash is public, data never leaves the device; (3) a genuinely-useful free tier that funnels to full storage.
**Open questions / risks (must resolve before build):** (a) **MESSAGING = brand risk #1** — "proof, not preservation"; a user must NEVER mistake an anchor for a backup (lose the local file = gone). (c) **Sync semantics** — an anchor-only file has no on-chain data, so it can't sync/download to another device; decide per-file toggle vs per-drive mode vs mixed. (d) **Interop / build-vs-lean** — see ar-io-anchor note below. (e) **Verification flow** — re-hash the local file → compare to the on-chain hash → show verified/timestamp. Post-beta; capture now, spec later.
**`ar-io/ar-io-anchor` (researched 2026-07-05 — https://github.com/ar-io/ar-io-anchor):** ar.io's official TS SDK is the *write path* of their verification stack and does EXACTLY this — "an anchor is a permanent on-chain record proving data existed at a specific time"; **raw data is never uploaded, only a hash + metadata**, hash computed locally, uploaded via Turbo as a signed ANS-104 event envelope (`ario.events/v1`); user gets a txId as proof. API: `createAnchorer()`, `anchor({data, ref})` → receipt with txId, and **`batch()` — a Merkle batcher that anchors many events under ONE on-chain checkpoint with per-event offline-verifiable inclusion proofs** (so anchoring a whole drive ≈ one tx, ~free). Verification is the separate read-only `@ar.io/proof` package (offline inclusion proofs, third-party-verifiable). Deps minimal (@ar.io/proof, @noble/hashes, @noble/ed25519; Turbo via fetch). **This confirms feasibility and gives us a foundation to lean on rather than invent.** KEY DESIGN DECISION it forces: ar-io-anchor uses ar.io's OWN event-envelope format, NOT ArFS — so decide (i) use ar-io-anchor's envelope (interoperable with ar.io's proof/verification stack + free batching, but an anchor is NOT an ArFS file entity → won't appear as a file in ArDrive/ardrive-web drives), (ii) build anchor-as-a-data-less-ArFS-file (shows in the drive, but outside ar.io's verification stack), or (iii) hybrid — an ArFS file entity that links an ar-io-anchor proof. The batching win strongly favors leaning on the package for the proof primitive; the "shows up in your drive" UX favors an ArFS wrapper. Resolve in the post-beta spec.

### FEAT-5 · P2 · Track B · `idea` (post-beta)
**Auto-classification / smart tagging (Phil, 2026-07-05; D-029).** As files sync, analyze content and attach tags/categories (image scene/object, OCR text, document type — receipt/invoice/ID, screenshots) for search + organization — the iCloud/OneDrive "smart" experience. **The differentiating angle: ON-DEVICE / local-first** — classification that NEVER sends your data anywhere (iCloud/OneDrive process in the cloud), which is exactly ArDrive's privacy+ownership ethos. Runs a small local model (ONNX/transformers.js + OCR) in Electron. Value: content search ("receipts", "screenshots from 2025") without manual folders; modern-file-manager parity; a genuine "private smart tagging" story; richer ArFS metadata.
**Key design decisions (must resolve before build):** (a) **On-device, not cloud** — mandatory for the ethos; cost = bundling a model + cross-platform inference (installer size, CPU/GPU variance, battery). (b) **Local-mutable tags by DEFAULT, opt-in to publish on-chain** — permanence makes on-chain auto-tags a PERMANENT commitment (can't cleanly un-tag; exposes your files' classification publicly on public drives) and a mis-tag is forever; keep tags in the local SQLite (re-computable, private) and let the user opt-in to write confirmed tags as ArFS custom tags. (c) **Suggestions, not silent truth** — auto-tags are user-confirmable, never auto-mis-file a sensitive doc. (d) **Search/filter UI** + an ArFS tag schema. Bigger than FEAT-4 — a whole feature track; post-beta. NOTE a cheap near-term subset exists: basic *type* grouping (images/docs/video by mime/extension) is non-ML and could be small beta-polish; the content-aware ML is the post-beta feature.
